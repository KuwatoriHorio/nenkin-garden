# タスク: analysis-002 — 砂糖源スナップ規則の修正（孤立ビーコンスパイク対策）

`loop-engineering-rules-v0.md` §11 のフォーマットに沿ったタスク文。
**前提**: `core-001` の §8 エスカレーション（iter:4）で、砂糖源→最近傍ノード snap が
**孤立ビーコンスパイク**（骨格上 size=1 の成分）を拾い、近傍に本体網があっても
別成分と判定される、と診断された。core は変更しない（core ← analysis の一方向依存）。

---

## 位置づけ
`analysis/flow.rs` の端子接続と連結判定を見直す。現状は各砂糖源を「唯一の最近傍ノード」へ
tap しており、ビーコン細胞が孤立ノード化するとそこに吸着してしまう。これを
**近傍の実ネットワーク（半径内の骨格ノード群）へ tap** し、source/sink の連結は
**拡張グラフ（骨格エッジ + tap エッジ）上の実接続**で判定するように改める。

## task_id
`analysis-002`

## goal
砂糖源の端子接続を「半径 `tap_radius` 内の骨格ノードへの複数 tap（無ければ最近傍1点に
フォールバック）」へ変更し、source/sink 連結を拡張グラフの union-find で判定する。
これにより、孤立スパイクの近傍に本体網があれば正しく連結と判定される。
core・決定性・非侵襲は不変。

## acceptance_test（headless で判定可能）
1. **決定性/非侵襲**: 同一 State から解析2回で指標完全一致。解析前後で `state_hash` 不変。
   core は無変更。
2. **近傍網への接続（修正の主眼）**: 制御シナリオ「本体網 + その半径内に置いた砂糖源
   （ビーコン孤立スパイク）」で、修正後は `flow_connected=true`・`total_conductance>0`
   （修正前は false だった）。
3. **過剰連結の防止**: `tap_radius` を超えて離れた2つの独立網に置いた2砂糖源は
   `flow_connected=false` のまま（faint な近接だけで繋げない）。
4. **グラフ整合の維持**: `num_cc` が core の `num_cc`（同一しきい）と一致し続ける
   （snap 規則変更は骨格グラフの成分数に影響しない）。
5. **リグレッション**: `core-000`/`analysis-001` の既存テストが緑。tap 規則変更で値が動く
   golden（例: 制御連結テストの実効抵抗）は**意図変更として理由付きで更新可**（§5/§7）。

## constraints
- **core を変更しない。** 変更は `analysis/`（主に flow.rs）と params のみ。State は読むだけ。
- 決定性契約（§2）維持: ノード/tap の走査順を正準化し、結果が揺れないこと。
- `tap_radius` 等は params 集約（ハードコード禁止）。**過剰連結を防ぐ小さめの半径**にする
  （§7: 「見かけ上つなげる」しきい恣意化はしない。連結は拡張グラフの実接続で判定）。
- 迷い・設計軸に触れる判断が要れば §8 で停止して提案。
