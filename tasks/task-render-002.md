`loop-engineering-rules-v0.md` §11 のフォーマットに沿ったタスク文。
**前提**: `analysis-001/002/003`（効率ネットワーク解析）完了。設計メモ §2 の `render/` レイヤ。

## 位置づけ
現在の可視化（`render_svg`）は trail **密度**のヒートマップ＋骨格ピクセル。本タスクは
同じ State から導いた**グラフ構造と流量**を静的 SVG で描く別レンズ（a 案）。
`render/` は core と analysis を**読むだけ**（core ← analysis ← render の一方向, 逆依存禁止）。
シミュレーションは不変・非侵襲。WASM デモへの統合は対象外（render-003 候補）。

## task_id
`render-002`

## goal
`analysis::analyze` の出力（グラフ・MST・流量）から、ノード/エッジのグラフを静的 SVG に
描画するエクスポータ（例 `render_graph_svg` bin, 出力 `docs/network_graph.svg`）を実装する。
描画要素:
- **ノード**（分岐点・端点・砂糖源）を点で、**エッジ**を線で。
- **幹線＝エッジ太さを流量 |I_e| に対応**（source/sink 間の主経路が太くなる）。
- **MST を実線・冗長辺（ループを閉じる辺）を破線**（redundancy の可視化）。
- **連結成分を色分け**（num_cc / largest_cc が区別できる）。
- **エッジ標高で色付け**（低地を這う様子＝edge_mean_elevation）。任意。
- 砂糖源を source/sink として強調。
背景に陸海の薄い下地を敷いてよい（density ビューと重ねられる下地）。

## acceptance_test（headless で判定可能）
1. **決定性/一意性**: 同一 State から2回生成した SVG がバイト一致。同一 `state_hash` → 同一図。
2. **非侵襲**: 生成の前後で `state_hash` 不変。core・analysis のロジックは変更しない
   （エッジ電流の**公開**追加を除く。下記 constraints）。
3. **グラフ整合**: 生成物に描かれるノード数・エッジ数が `analysis` の `nodes`/`edges` と一致
   （SVG に `data-nodes`/`data-edges` 等のメタを持たせ数値で検証、または要素数で検証）。
4. **幹線写像の純関数テスト**: 流量 |I_e| → 線幅 の写像が単調（流量最大のエッジが最大幅、
   0 流量は最小幅）であることを native で検証する。
5. **出力**: `docs/` に SVG を出力し、GitHub Pages で読める。トップページから導線を張る。
6. `core-000`/`analysis-001/002/003` の既存テストが全緑（不変条件維持）。

## constraints
- **core を変更しない・逆依存禁止**（設計メモ §2）。State は読み取り専用。
- **流量可視化のための analysis 追加は最小限・読み取り専用**: 現状 `flow::solve` は per-edge
  電流 `I_e`（と必要なら node 電位）を内部で計算して破棄している。これを `AnalysisResult`
  に**公開するだけ**の追加は可（State 非書換・決定性維持・既存指標は不変）。それ以外の
  analysis アルゴリズム変更はしない。
- 決定性契約（§2）維持: ノード/エッジ/成分の描画順を正準化し、SVG がバイト一致すること。
- 色・線幅・しきいは可能な限り params/定数に集約（ハードコード最小化）。
- §7: 見栄えの自己申告で合格としない。合格根拠は上記の判定可能条件（手動確認は補助）。
