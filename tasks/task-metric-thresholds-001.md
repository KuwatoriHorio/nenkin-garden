# タスク: metric-thresholds-001 — メトリクスしきいの再調整（分解能と退化の是正）

`loop-engineering-rules-v0.md` §11 のフォーマットに沿ったタスク文。
**前提**: `calib-001` の実測で、いくつかのメトリクスが分解能不足／退化していると判明。

---

## 位置づけ（calib-001 の観測）
- `elev_trail_ratio` が代表シナリオで**常に 0**（高標高帯 E≥E_hi に網がそもそも届かず
  `mean_trail_hi=0`）。忌避の健康診断としての情報量がない＝退化している。
- `coverage`/`max_cc`/`num_cc` はしきい `θ_cov, θ_cc` に依存。値が「全か無か」に
  潰れず、変化を検出できる分解能を持つべき（現状の分布が妥当か検証する）。

メトリクスしきい `θ_cov, θ_cc`（および健全性指標の定義）を実測分布に基づき再調整し、
メトリクスが変化を意味あるレンジで捉えるようにする。**不変条件の定義は人間所有**で不可侵。

## task_id
`metric-thresholds-001`

## goal
`θ_cov, θ_cc` を代表シナリオの trail 分布に基づき再設定し、`coverage` と連結成分系
（`num_cc, max_cc`）が有意な分解能を持つようにする。あわせて忌避の健康診断を、
退化しない指標へ見直す（例: `mean_trail_hi/mean_trail_lo` の扱い、または E_hi 帯の
定義・重み付け）。`analysis` は `θ_cc` を core と共有する整合を維持する。

## acceptance_test（headless で判定可能）
1. `calibrate` 出力で、`coverage` と `num_cc/max_cc` の分布が単一値に潰れておらず
   （relMAD > 0）、しきい変更の前後で分解能が改善（または維持）していることを示す。
2. 忌避の健全性指標が退化していない: 代表シナリオで健全性シグナルが 0 固定でなく、
   かつ Tier 0 不変条件 `mean_trail_hi < mean_trail_lo` を破らない。
3. `analysis` の `num_cc` が core の `num_cc`（同一 `θ_cc`）と一致し続ける。
4. `core-000`/`analysis-001` の既存不変条件・受け入れテストが全て緑のまま。

## constraints
- **§7 完整性**: しきい変更は「メトリクスの分解能改善」が目的。テストを通すために
  都合よく緩めない/潰さない。不変条件のしきい（人間所有）は編集不可。
- `θ_cc` は core と analysis で共有される前提を壊さない（受け入れ #3）。
- 決定論契約（§2）を崩さない。変更は params に集約（ハードコード禁止）。
- 迷い・矛盾があれば §8 で停止して提案（健全性指標の再定義は設計軸に触れうる）。

- 新しい analysis 指標の追加（必要なら `analysis-002` として別途）。
