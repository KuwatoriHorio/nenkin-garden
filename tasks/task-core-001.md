## 背景（なぜこのタスクか）

`analysis-001` 実装時に、既定シナリオでは core が育てる trail 網が断片化しており
（seed42/160tick で `num_cc=56`, `largest_cc=2`）、2 つの砂糖源が別々の連結成分に
落ちるため `flow_connected=false`（実効抵抗 ∞）になることが分かった。
これはトポロジ上は正直な結果だが、「砂糖源間の流れやすさ（conductance / 輸送効率）」を
測る analysis の主目的が既定シナリオでは発揮されない。

そこで **core 側のパラメータ既定値を調整**し、代表シナリオで 2 砂糖源が同一連結成分に
入って `flow_connected=true` かつ conductance が有限・有意になる網を作る。
`analysis/` は変更しない（core を読むだけの契約は不変）。

規約 §11: パラメータの既定値は編集可（受け入れテストが守る範囲で）。
不変条件・ゴールデン・既存受け入れテストは人間所有＝編集不可。

## task_id
`core-001`

## goal
core のパラメータ既定値（`src/params.rs`）を調整し、下記の代表シナリオで
2 つの砂糖源が同一連結成分に接続され、`analysis` の `flow_connected=true` かつ
`total_conductance > 0` となる網を生成する。`core-000` の決定性・不変条件と、
既存の全受け入れテストを維持したまま行う。

## 代表シナリオ（人間定義・固定）
- 世界: 既定の合成仮想列島（`make_synthetic_archipelago`）。
- 砂糖配置: 低標高の陸セル基準点 `(sx, sy)`（既存テストと同じ選び方＝
  `land && E<E_lo` の中央値セル）に、
  - `tick=0`: `place_sugar {x:sx,   y:sy,   strength:400}`
  - `tick=5`: `place_sugar {x:sx+6, y:sy+4, strength:300}`（距離約 7.2 セル）
- 実行 tick 数は tuner の裁量で増やしてよい（ただし §7: テストを通すためだけの
  極端な増加は不可。性能予算 §4 を守る）。既定の代表 tick 数も併せて定義すること。

## acceptance_test（headless で判定可能）
1. **連結**: seeds=[1,42,1337] の各終端 State に対し `analysis::analyze` を実行し、
   `flow_connected=true` が **複数シード中央値で成立**（=3 シード中 2 つ以上で true）。
2. **有限で有意なコンダクタンス**: 上記で連結したシードにおいて
   `total_conductance > 0` かつ有限、`effective_resistance` が有限・正。
3. **輸送効率が有効**: 連結シードで `0 < transport_efficiency ≤ 1`。
4. **core 整合の維持**: `analysis` の `num_cc` が core の `num_cc`（同一しきい値）と
   一致し続ける（`analysis-001 #3` を壊さない）。
5. **リグレッション無し**: `core-000`（不変条件・再現性・ソフト標高忌避）と
   `analysis-001`（#1〜#6 + 流れソルバ検証）の**既存テストが全て緑のまま**。
6. **メトリクス許容範囲（規約 §4）**: 代表シナリオで `coverage` と `largest_cc` が
   基準（本タスク着手前の値）から悪化しない（連結性を上げる変更なので改善方向のはず）。

## constraints
- **編集可**: `src/params.rs` の既定値、代表シナリオ定数、必要なら
  `src/bin/*`・テスト内シナリオのヘルパ（ただし**既存の受け入れアサートは変更不可**）。
- **編集不可（人間所有・§7/§11）**: 不変条件の定義としきい値、`core-000`/`analysis-001`
  の受け入れテストのアサート、決定性契約（§2）、プロダクト設計軸（§0）、
  `analysis/` の core 非侵襲性（core ← analysis の一方向依存）。
- **決定性維持**: 単一 PRNG・index 昇順・スナップショット感知（§4）を崩さない。
- **過学習禁止**: 単一シードだけで連結する調整は不可。必ず複数シード中央値で判定（§7）。
- **完整性（§7）**: テストを通すために ticks/seeds/解像度を都合よく縮小しない。
  しきい値 `theta_cc/theta_cov` を下げて見かけ上つなげる場合も、
  不変条件・ソフト忌避・core 整合を壊さない範囲に限る（安易なしきい値緩和で
  "連結に見せる" のは禁止）。
- **性能予算（§4）**: `tick_ms` が基準比 +10% を大きく超えない。ticks 増加や密度増加で
  実行時間が膨らみすぎないこと。

## 実装者(Claude Code)の裁量・調整レバー候補
以下は候補であり、どれをどう動かすかは裁量（1 タスク=最小の仮説, §6）:

- **実行 tick 数**: 網が成長・連結するまでの時間を確保。
