## 背景（なぜこのタスクか）
ユーザー観察（2026-07-12・demo-net）: 現状の探索は**全体として「線が伸びる」**ように見える。各前線ノードが
持続方向へ 1本ずつ伸長するため、面的に開かず糸状に伸びる。**探索時は周囲へ扇状に拡散して広がってほしい**
（実際の Physarum の「数箇所から扇状に広がる」に近づける）。**ノード数が増えてもよい**。まず**この探索方法に
変えて挙動を観察したい（試行）**。＝ exploratory: 実装後に demo-net へ反映して**目視で挙動を確認**し、その結果を
見て既定値の採否・微調整を人間が判断する。

## task_id
`netphys-004`

## goal
`netphys_step` の **Phase1 探索を「単一方向の線的伸長」から「扇状拡散」に変える**。各前線ノードから**周囲の
角度範囲へ複数の probe を扇状に**広げて枝分かれし、探索が**面的に開く**ようにする（ノード数増を許容）。
通常予算 regime（`NetParams::default()`）で**探索が線でなく面的に広がる**ことを headless で実証し、demo-net で
目視確認する。ソフト（壁なし）・保存則・決定性を保ち、netphys-001 ①②⑤⑥・netphys-002 ③④・既存
Jones/tree/analysis 全テストを**緑のまま**にする。

## 実装方針
- **扇状拡散**（`src/netphys/step.rs` Phase1・`src/netphys/state.rs` の `NetParams`）:
  - 前線ノードの伸長を、**持続方向±角度範囲に複数 probe を張る扇状探索**へ変える（生存 probe ごとにノード/辺を
    追加＝枝分かれ）。1本しか通らない場合は従来どおり単枝。**面的に開く**ことを狙う。
  - 新規パラメータ例 `fan_count`（1本あたりの probe 本数）・`fan_spread`（扇の半角）・分岐確率。既定は**扇状に
    広がる妥当値**を探索用一時テストで実測。既定を保守値（`fan_count=1`）にすれば現状に縮退できる（後方互換）。
  - **ノード数増を許容**: 必要なら `node_cap`/`edge_cap` の既定を上げてよい（＝設計判断であり、テストを通すための
    水増しではない・§7）。ただし**⑥有界は維持**（cap 内・爆発しない・周期 Kirchhoff が予算内）。
- **壁禁止（§0）**: 海/範囲外の probe は従来どおり棄却（ハード境界）。標高は既存のコスト/実効抵抗のまま（本タスクでは
  標高方向バイアスは扱わない＝netphys-003）。
- **保存則・決定性を壊さない**: 分岐数・方向の変更のみで質量会計（free↔D·L・回収/消費/刈り込み）は不変。
  扇状分岐でも `total_mass == collected − consumed` を締める（⑤が引き続き検証）。単一 PRNG・id 昇順を厳守。

## acceptance_test（headless・複数シード中央値・S9 部分集合 [1,42,1337]・新規 `tests/netphys_004.rs`）
1. **扇状拡散＝面的に広がる（主判定）**: 固定 tick 時点で、探索が**線状でなく面的に開く**ことを定量で締める。
   例: 初期数十ノードの**方向分散（角度カバレッジ）／占有グリッドセル数／凸包面積**が、扇状無効
   （`fan_count=1`＝現状の線的伸長）baseline より**中央値で有意に大きい**。
   - baseline 比較で締める（§7 exemplary を推奨）: `fan_count=1`（線）では面的指標が小さいことを**赤**として確認
     → 扇状既定（効く値）で**緑**。同一 world・同一 seed で対比。
- **回帰（必須）**: `tests/netphys_001.rs` ①②⑤⑥・`tests/netphys_002.rs` ③④が**緑のまま**。
  特に ⑤（保存則/決定性/境界/scarce 忌避）・③（前進波・free_budget>50）・④（効率）・⑥（有界＝cap 内）を壊さないこと。
  Jones/tree/analysis/render-wasm 全テストも緑。**決定性 hash 契約**を壊さない（同一 seed・入力→同一 `netphys_state_hash`）。
- パラメータ（`fan_count`・`fan_spread` 等）は**探索用一時テスト**で実測 → 決定後に削除。
  まず①（fan_count=1 で線的＝面的指標が小）の赤を確認してから実装。
- **試行としての目視確認（合否外だが必須の締め）**: 全テスト緑を確認後、**demo-net の wasm を再生成**して
  ライブで「扇状に広がる」挙動を人間が目視確認できる状態にする（デモの JS 描画ロジックや netphys 以外は無変更・
  wasm 再生成のみ）。目視結果を見て既定値の採否・微調整を人間が判断する。

## §8 停止の指針（このタスク特有）
扇状化は有界性・保存則と綱引きになりうる。次に当たったら**§7 のごまかしをせず停止**し、成立した部分と
詰まり・選択肢を報告する:
- 扇状化でノード/辺が cap を突く・爆発する／周期 Kirchhoff が予算超過（⑥有界 vs 面的拡散の綱引き）。
- 面的に広げると前進波③が止まる／効率④が悪化する（枝が分散して背骨に集約できない）。
- 分岐会計で保存則⑤が締まらない。
無理に①を通すために cap/tick を縮小したり（cap 引き上げは可・水増しは不可）、netphys-001/002 の受け入れを
弱めたりしない。設計軸（§0）自体を変えたくなったら §8 で停止・相談。
