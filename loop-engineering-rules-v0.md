# 粘菌ガーデン — 自律ループ・エンジニアリング規約 (v0)

自律エージェントが「コード変更 → テスト → 修正」を反復するための契約書。
エージェントはこの規約に**常時従う**。規約自体の変更は人間の承認タスクでのみ行う。

---

## 0. 前提となるプロダクト仕様（変更不可の設計軸）

- モデル: エージェントベース（Jones模型）。trailマップ + 拡散・減衰。
- ジャンル: 園芸・観察型。プレイヤーの動詞は「砂糖を置く／取り除く」「時間速度の変更」のみ。
- 場: 陸海マスク + 標高場 E（仮想日本列島）。
- 標高忌避: **ソフト**（感知ペナルティ + 傾斜コスト + 定着差 + 減衰差）。壁は作らない。
- 成長ループ: 砂糖到達 → バイオマス増 → エージェント上限増 → さらに拡張。
- 実装方針: **決定論コア（シード固定・描画分離・ヘッドレス実行可）を「正」とする。**

これらは「面白さの正体」なので、ループが勝手に方針転換してはならない。
変更したい場合は §8 に従い停止して人間に諮る。

---

## 1. 目的と原則（Prime Directive）

エージェントの仕事は次の3つを**同時に**満たすこと。順序は優先度。

1. **不変条件を絶対に壊さない**（§3 のハードゲート）。
2. **合意済みメトリクスを許容範囲を超えて悪化させない**（§4）。
3. **与えられたタスクの受け入れテストを通す**（§9・§10）。

「良くなった」の判断はエージェントの主観ではなく、**人間が書いた受け入れテスト**が定義する。
見た目の良し悪しをエージェントが自己申告して合格にすることは禁止（§7）。

---

## 2. 決定論コアの契約（ループの土台）

- コアは純関数として振る舞う: `step(state, inputs, params, rng) -> state'`。
  同一 (初期state, inputs列, params, seed) は**必ず同一の最終stateとハッシュ**を生む。
- 乱数は**単一のシード付きPRNGのみ**。壁時計・実時間・ハッシュマップの反復順など
  非決定的な入力を状態遷移に使わない。
- 描画は state を**読むだけ**。描画コードは state を書き換えない。
- ヘッドレス実行エントリを持つ:
  `run_headless(seed, input_script, ticks) -> {metrics.json, final_state_hash}`。
- 浮動小数の非決定性を避けるため、対象プラットフォーム/ビルドを固定して比較する。

この契約が崩れると全テストの信頼性が消える。コアの決定性を壊す変更は
「機能追加」ではなく**リグレッション**として即revert対象。

---

## 3. 不変条件（ハードゲート・毎イテレーション実行）

すべてのイテレーションで、複数シードに対して検証する。**1つでも破れたら変更は破棄。**

- **有限性**: trail場・エージェント座標に NaN / Inf が出ない。
- **保存則**: バイオマス = 回収砂糖累計 − 消費。負にならない／自然増殖しない。
- **境界**: エージェントは海マスク外へ出ない。全座標が場の範囲内。
- **再現性**: 同一シード・同一入力列 → 同一 final_state_hash（描画OFFでも一致）。
- **標高忌避（ソフト）**: 高標高帯の平均trail密度 < 低標高帯の平均trail密度。
  ゼロではなく「有意に低い」で判定（ソフト忌避のため）。

不変条件のコードとしきい値は**人間所有**。エージェントは編集不可（§7・§11）。

---

## 4. メトリクスと許容範囲（ソフトゲート）

各イテレーションで headless 出力から取得し、**基準（前回グリーン）から劣化していないか**を判定。
しきい値は **calib-001（2026-07-07, 実測: seeds 1..=64 / 160tick / 代表シナリオ）で確定**。
根拠は単一シード relStd から見積もる「9本中央値の標準誤差 ≈ relStd×0.417」。固有ノイズで
誤検知しない幅に設定した（旧初期値 −2%/−5% は過敏で常時誤検知していた）。再測定は
`cargo run --bin calibrate` で可能。

| メトリクス | 意味 | 判定（方向つき, 中央値比較） | 根拠(実測) |
|---|---|---|---|
| 被覆率 | 陸地に対する網の到達面積 | 基準比 **−8%** 以内 | relStd 19% → SE≈8% |
| 回収砂糖レート | 単位時間あたり回収量 | 基準比 **−18%** 以内 | relStd 42% → SE≈18%（高分散） |
| 最大連結成分サイズ | 網の一体性 | 基準比 **−18%** 以内 | relStd 40% → SE≈17%（高分散） |
| 連結成分数 | 断片化の指標 | 増加 **+10%** 超で要注意(warn) | relStd 8%（安定） |
| 忌避の健康診断 | 網が低標高に偏るか | ハード=不変条件 mean_hi<mean_lo(§3)。ソフト= **elev_avoidance** 基準比 **+8%** 超で悪化(warn) かつ mean_trail_lo −12% 以内 | metric-thresholds-001: elev_trail_ratio は退化(常に0)のため trail 加重平均標高/陸地平均 の比 elev_avoidance へ変更。med≈0.13, relStd 5.4% → SE≈2.3% |
| 1ティック処理時間 | 性能退行 | 基準比 **+25%** 以内（性能予算） | relStd 28%・裾重い・機械依存 |

### 複数シード中央値の集計法（calib-001 で確定）
- **正準シード集合 S（奇数9本）** = `{1, 7, 13, 42, 99, 256, 1337, 2024, 31337}`。
  Tier 0 不変条件は S 全9本で pass 必須。Tier 2 メトリクスは S 上の中央値で比較。
  （安定メトリクスの中央値は k≈9 で大標本値に一致。奇数=中央値が一意。）
- 手順: S で headless →各 (metric, seed) を算出（決定論）→ **S 上の中央値**（平均は使わない
  ＝外れ値耐性）→ 基準（前回green）中央値と方向つき比較:
  - 低いほど悪い（被覆/砂糖/max_cc/mean_trail_lo）: `candidate < base×(1−tol)` で退行。
  - 高いほど悪い（num_cc/tick_ms/elev_ratio）: `candidate > base×(1+tol)`（ratio系は `> base+ε`）で退行。
- ゲート = ハード対象メトリクスが**全て**許容内（AND）で pass。num_cc 増・ratio 漸増は
  warn（表面化するがブロックしない、不変条件を破らない限り）。
- **単一シード最適化＝過学習を防ぐ**（§7）。シード集合を変える場合は常に奇数を保ち、
  golden を理由付きで貼り直す。

## 5. テスト階層

- **Tier 0 — 不変条件**（§3）: 毎回・複数シード・安価。ハードゲート。
- **Tier 1 — ゴールデン/スナップショット**（粒度は calib-001 で確定）:
  正準シード集合 S + 固定入力スクリプトに対し、**主= `final_state_hash` をシードごとに**
  保存（完全一致・最大感度: 量子化 state が 1ULP 変われば反転し、あらゆる挙動変化を検出）。
  **副= metrics ベクトル**（各シード値と S 上の中央値）を §4 の許容%付きで併記
  （hash が「変わった」ことしか示さないのに対し、変化の**量と方向**を人間可読に与え、
  意図変更時の golden 更新判断に使う）。両立で「厳密検出＋解釈可能」。
- **Tier 2 — メトリクス/プロパティ**（§4）: 傾向の健全性（被覆は時間で増える、
  忌避の順序が保たれる、保存則 等）。
- **Tier 3 — 性能**: ティック時間が予算内。

**ゴールデンの更新は「意図した挙動変更」のときだけ**、差分理由をログに残して行う（§7）。

---

## 6. ループ手順（1イテレーションの厳密な流れ）

1. **タスク取得**: 未完了タスクを1つだけ取る（§11 の形式）。範囲外に手を出さない。
2. **失敗するテストを先に書く**: 機能なら受け入れテスト、バグなら再現テスト（赤を確認）。
3. **最小の変更**: 1タスク=1仮説。無関係なリファクタを混ぜない。
4. **ヘッドレス実行**: Tier 0（全シード）→ Tier 1 → Tier 2 → Tier 3。
5. **評価**: 不変条件（ハード）＋メトリクス許容範囲（ソフト）＋タスク受け入れテスト。
6. **判定**:
   - 全通過 → 採用（keep）。ゴールデン更新が必要なら理由付きで更新。
   - 失敗 → revert して 3 から再試行。**同一タスクの試行は上限 N=3 回**（calib-001 で確定。
     実タスクは各 ≤1 回の是正で採用。決定論テストは失敗信号が明確で、3 超の反復は
     仕様/前提の問題＝§8 で人間に諮る方が妥当）。
   - 上限到達 → 停止して人間へエスカレーション（§8）。
7. **記録**: イテレーション記録を残す（§10）。
8. **次へ**: 停止条件に触れていなければ 1 に戻る。

**各イテレーションは常に revert 可能な単位**（コミット単位）で行う。

---

## 7. 完整性ルール（テストのごまかし禁止・最重要）

以下はすべて**即時失格・自動revert・人間へ報告**の対象。

- 不変条件・ゴールデン・受け入れテスト・しきい値の**無効化・削除・緩和**。
- ゴールデン値を「意図した挙動変更」の理由なしに書き換える。
- ティック数・シード数・場の解像度を**テストを通すために**縮小する。
- 特定シードだけで良くなる変更（過学習）。※必ず複数シード中央値で判定。
- 例外の握りつぶし・NaNのクランプ等で不変条件チェックを実質無力化する。
- 「見た目が良くなった」等の**主観の自己申告**で受け入れとすること。
- 性能予算を通すために計算をスキップして**挙動を変える**こと。

原則: **テストを事実に合わせるのではなく、事実（挙動）をテストに合わせる。**
テストが間違っていると考える場合は、勝手に直さず §8 で停止・提案する。

---

## 8. 停止条件とエスカレーション

次のいずれかで**ループを止め、人間に判断を仰ぐ**（勝手に進めない）。

- 同一タスクで試行上限（N回）を使い切っても受け入れテストが通らない。
- 満たせない／矛盾する不変条件・要件に突き当たった。
- タスクの仕様が曖昧で、複数の妥当な解釈がある。
- 性能が予算を大きく超え、挙動を保ったままでは戻せない。
- **不変条件・ゴールデン・プロダクト設計軸（§0）自体を変えたくなった**とき。
- §7 のごまかしをしないと通せない状況。

エスカレーションには「何を試したか・なぜ詰まったか・選択肢」を添える。

---

## 9. 受け入れテスト（タスク完了の定義）

- 各タスクには**人間が書いた／承認した受け入れテスト**が付く。これが唯一の「完了」定義。
- 受け入れテストは決定論コア上で headless に判定可能でなければならない
  （例: 「砂糖を座標Xに置き200ティック後、Xを含む連結成分の被覆が閾値以上」）。
- エージェントは受け入れテストを**編集しない**。不足を感じたら §8 で提案する。

---

## 10. イテレーション記録フォーマット

毎イテレーション、以下を追記（再現性と監査のため）。

```
- iter: <連番>
  task: <タスクID>
  hypothesis: <この変更で何を良くするつもりか 1行>
  diff_summary: <触ったファイル/関数の要点>
  seeds: [<検証シード>]
  invariants: pass|fail(<どれ>)
  metrics: { coverage: Δ, sugar_rate: Δ, max_cc: Δ, tick_ms: Δ, ... }
  goldens_updated: none | <ファイル: 理由>
  decision: keep | revert
  note: <あれば>
```

---

## 11. スコープ（触ってよい / だめ）

- **編集可**: シミュレーションコア実装、描画、パラメータの既定値（受け入れテストが守る範囲で）。
- **編集不可（人間所有）**: 不変条件の定義としきい値（§3）、ゴールデンの期待値（意図変更時に理由付きでのみ）、受け入れテスト（§9）、本規約、プロダクト設計軸（§0）。
- ネットワークは使わない前提。外部依存の追加は §8 で提案。
